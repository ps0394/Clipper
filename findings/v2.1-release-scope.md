## 1. Why This Release Exists

Sessions 8 / 9 / 9.5 established that Clipper's `parseability_score` and `universal_score` composites are calibrated on corpus-002 and **do not generalize** to corpus-003 (Pearson r vs. judged accuracy ≈ +0.10 against a ship gate of r ≥ +0.35). This is not "the tool is broken" — the per-pillar measurements (semantic HTML, Schema.org, axe-core a11y, RFC 7231 negotiation, Readability extractability, metadata completeness) are real, well-defined signals against published standards. What does not generalize is the **weighted composite** that turns those pillars into a single 0–100 number with "agent-ready" bands.

Clipper-next (post-v3) is a separate, larger effort scoped in `findings/post-v2-roadmap.md` §10.3. That work will not ship soon. In the meantime, the current `main` branch is the only Clipper anyone has, and current docs/CLI present the composite scores with confidence the data does not support.

**v2.1 closes that gap by re-labeling, without removing functionality.** Existing users keep working. New users see honest claims.

## 2. Goals

1. **Truthful framing.** README and scoring docs state what corpus-003 showed: composite generalization failed, pillar measurements are intact.
2. **No silent breakage.** All existing CLI commands, JSON schemas, and output files keep working byte-for-byte.
3. **A safer default for serious users.** Add a `--diagnostic-mode` flag that emits pillar-level + raw-signal output and suppresses the composite scores. This is the mode the data supports for cross-page comparison today.
4. **Clear "what this is / what this isn't" boundary.** Prevent v2.1 from being mistaken for a methodology revision.

## 3. Non-Goals

- **No** new pillars, weights, or scoring models.
- **No** removal of `parseability_score` / `universal_score` from existing JSON output (would break scripts).
- **No** change to evaluator internals (`access_gate_evaluator.py`, `parse.py`, `crawl.py`, profile weights).
- **No** new corpus, no Phase 5 work, no Foundry calls.
- **No** GitHub release / PyPI packaging changes (out of scope; release this as a tagged commit on a branch).

### 4.2 CLI

Add one flag to `python main.py express`:

```
--diagnostic-mode
```

Behavior when set:
- Output JSON keeps the same schema, but `parseability_score` and `universal_score` fields become `null`.
- Adds a per-result `diagnostic_mode: true` field for downstream filtering.
- Markdown report **prepends a "Diagnostic mode is ON" banner** at the top, above the existing executive summary. Composite-scored sections of the report still render (averages and sort orders treat null composites as 0.0 to keep the report end-to-end renderable) but the banner makes the suppression explicit. This is less surgery than the original scope (which proposed omitting the score banner entirely) and preserves the structural sections users rely on.

When **not** set: identical behavior to today, *plus* a one-line "Methodology disclosure" banner at the top of the markdown report linking to the post-v2 roadmap. (No JSON schema change beyond §4.3.)

Implementation: a few lines in `retrievability/cli.py` and `retrievability/report.py`. No changes to `score.py` math.

### 4.3 JSON schema annotation

Add a `methodology` block to **every score result** in the output JSON, present whether or not `--diagnostic-mode` is set:

```json
{
  "url": "https://...",
  "parseability_score": 65.6,
  "universal_score": 62.1,
  "component_scores": { ... },
  "methodology": {
    "scoring_version": "v2-evidence-partial",
    "calibration_corpus": "corpus-002",
    "generalization_status": "fails on corpus-003 (Pearson r ≈ +0.10 vs ship gate +0.35); see findings/post-v2-roadmap.md",
    "recommended_use": "pillar-level diagnostics; not validated for ranking pages against each other",
    "release": "v2.1"
  },
  "diagnostic_mode": false
}
```

Note: the methodology block is **per-result, not top-level**. The current JSON root is an array `[…]` of results; making it top-level would change the root shape to `{…}`, which is a breaking change. Per-result keeps every consumer working unchanged and ensures the disclosure is visible on every page regardless of how the array is indexed.

This is the single biggest accountability win — anyone parsing the JSON sees the caveat without having to read docs.

## 9. Open Questions

1. **Do we want a `--legacy-mode` flag** that omits the new `methodology` block for users who want byte-identical output? Probably no — the methodology block is exactly the win and shouldn't be opt-out.
2. **Should `--diagnostic-mode` be the default** in v2.1, with the composite available behind a `--composite` flag? Argues yes (forcing honest defaults), argues no (silent behavior change for existing scripts). Recommend: not the default for v2.1; revisit for v3.
3. **Versioning of `methodology.scoring_version`.** Current `v2-evidence-partial` string is informal. v2.1 is a good moment to formalize it as part of the JSON schema and document the bump policy.
